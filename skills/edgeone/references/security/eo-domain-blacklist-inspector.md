# eo-domain-blacklist-inspector

用于查询 **EdgeOne 指定域名关联的安全策略**，识别哪个 IP 组承担黑名单拦截逻辑，并展示该 IP 组的具体条目。该技能的核心目标是：

- 获取指定域名当前生效的安全策略配置
- 从策略中定位承担黑名单拦截逻辑的 IP 组
- 查询该 IP 组的详细内容，输出完整的黑名单 IP 条目

## 适用输入

- 查一下 example.com 的安全策略里哪个 IP 组是黑名单
- 这个域名的拦截 IP 组是哪个
- example.com 现在用的是哪个 IP 黑名单
- 帮我看看 example.com 的 IP 封禁列表

## 前置条件

### 认证与 CLI 前置动作

1. 所有腾讯云 API 调用统一通过 `tccli` 执行。
2. 如果环境中尚未配置可用凭证，必须先引导用户执行 [auth.md](./reference/auth.md) 中说明的登录方式：

```sh
tccli auth login
```

3. 需要明确告知用户：执行 `tccli auth login` 后，终端会打印授权链接，并在用户完成浏览器授权前保持阻塞；授权成功后命令会自动结束。
4. 严禁向用户索要 `SecretId` / `SecretKey`，也不要执行可能暴露凭证内容的命令（尤其是 tccli configure list）。

### 业务前置条件

1. 在确认站点参数并执行后续查询前，必须先确认 `tccli` 已处于可用登录状态。
2. 在确认 `tccli` 可用后，再确认目标域名和对应的 `ZoneId`。
3. 用户必须明确提供目标域名（如 `example.com`）；如果用户只说"这个域名"而未给出具体值，需要先追问确认。
4. 本技能仅执行只读查询，不会对任何配置进行修改。

## 执行流程

### 1. 检查认证状态

优先确认当前环境中的 `tccli` 是否已登录且可用。

如果当前环境尚未登录或调用失败提示凭证不可用，先引导用户完成 [auth.md](./reference/auth.md) 中的浏览器授权登录，再继续后续步骤。

### 2. 确认目标域名与站点

在确认 `tccli` 可用后，再确认本次查询的最小必要信息：

- 目标域名（精确值，如 `example.com`）
- 所属站点的 `ZoneId`

如果用户未提供 `ZoneId`，先通过域名推断所属站点，或要求用户补充，再进入采集阶段。

### 3. 查询接口文档（按需）

如果对 `teo` 相关接口的参数、字段、数据结构或返回含义不确定，应按照 [ref.md](./reference/ref.md) 中的方法先检索文档，再执行 API 调用，不要臆测字段含义。

典型检索方式包括：

- 检索 `teo` 服务的接口列表
- 阅读对应 Action 文档确认参数
- 阅读数据结构文档确认字段含义

### 4. 顺序采集数据

围绕目标域名，按以下顺序采集数据：

**第一步：获取域名关联的安全策略**

```sh
tccli teo DescribeSecurityPolicy --ZoneId <ZoneId> --Entity <域名>
```

**第二步：获取站点下所有 IP 组列表**

```sh
tccli teo DescribeSecurityIPGroup --ZoneId <ZoneId>
```

**第三步：查询目标 IP 组的详细内容**

根据第一步和第二步的结果，定位到承担黑名单拦截逻辑的 IP 组 ID，再执行：

```sh
tccli teo DescribeSecurityIPGroupContent --ZoneId <ZoneId> --GroupId <GroupId>
```

如果接口返回提示参数不完整、权限不足、字段名不匹配或其他异常，应按以下顺序处理：

- 先说明失败原因
- 如果是参数不完整，先执行 `tccli teo <Action> help` 获取该接口所需参数；如果返回信息仍不够清晰，再执行 `tccli teo <Action> help --detail` 获取详细说明
- 如果仍有疑问，再根据 [ref.md](./reference/ref.md) 的方法查询对应接口文档
- 补充必要参数后重新调用

### 5. 识别黑名单 IP 组

在完成采集后，从安全策略中定位承担黑名单拦截逻辑的 IP 组：

- 在安全策略的规则列表中，查找动作为"拦截（Block）"且引用了 IP 组的规则
- 结合 IP 组名称（如包含 `blacklist`、`blocklist`、`deny`、`黑名单` 等关键词）辅助判断
- 如果存在多个 IP 组被用于拦截，逐一列出并说明各自的规则上下文

### 6. 生成结论

先输出结论，再给出支撑证据，结构参见"输出要求"。

## 数据采集与分析方法

### 1. 安全策略解析

基于 `DescribeSecurityPolicy`：

- 获取域名当前生效的完整安全策略
- 重点关注策略中引用了 IP 组的规则，尤其是动作为拦截/封禁的规则
- 记录每条相关规则的：规则 ID、规则名称、匹配条件、动作类型、引用的 IP 组 ID

### 2. IP 组列表核查

基于 `DescribeSecurityIPGroup`：

- 获取站点下所有 IP 组的名称、ID、描述信息
- 结合名称语义（如 `blacklist`、`blocklist`、`黑名单`）辅助识别用途
- 与安全策略中引用的 IP 组 ID 进行交叉比对，确认哪些 IP 组实际参与了拦截逻辑

### 3. IP 组内容展示

基于 `DescribeSecurityIPGroupContent`：

- 获取目标 IP 组中的完整 IP 条目列表
- 关注条目数量、IP 段范围、是否存在过于宽泛的 CIDR（如 `0.0.0.0/0`）
- 如果条目数量过多（如超过 100 条），优先展示前若干条并注明总数，避免输出过长

## 黑名单 IP 组识别规则

满足以下任一条件的 IP 组，应标记为**承担黑名单拦截逻辑**：

- 被安全策略中动作为"拦截/封禁（Block）"的规则直接引用
- IP 组名称包含 `blacklist`、`blocklist`、`deny`、`黑名单`、`封禁` 等语义关键词，且被任意拦截规则引用
- 规则描述或备注中明确说明该 IP 组用于黑名单拦截

如果存在多个 IP 组均满足上述条件，应全部列出，并说明各自对应的规则上下文。

如果无法从策略中明确判断哪个 IP 组是黑名单，应如实说明，并列出所有被拦截规则引用的 IP 组供用户人工确认。

## 输出要求

最终必须用中文输出，优先给结论，再给证据，结构固定如下：

### 1. 查询范围

- 目标域名
- 所属站点 `ZoneId`
- 查询时间
- 数据来源接口

### 2. 结论

直接给出最终判断，格式示例：

```
域名 example.com 的安全策略中，承担黑名单拦截逻辑的 IP 组为：

  - IP 组名称：blacklist-prod
  - IP 组 ID：ipg-xxxxxxxx
  - 关联规则：规则 ID xxx，动作：Block，触发条件：来源 IP 命中 IP 组
```

如果存在多个黑名单 IP 组，逐一列出。

如果未找到明确的黑名单 IP 组，说明"未在当前安全策略中发现承担黑名单拦截逻辑的 IP 组"，并列出所有被拦截规则引用的 IP 组供人工确认。

### 3. IP 组详细内容

列出黑名单 IP 组中的具体条目：

| 序号 | IP / CIDR | 备注 |
|---|---|---|
| 1 | 1.2.3.4 | ... |
| 2 | 10.0.0.0/8 | ... |

如果条目超过 20 条，展示前 20 条并注明"共 N 条，仅展示前 20 条"。

### 4. 安全策略关联规则摘要

列出安全策略中所有引用了 IP 组的规则：

| 规则 ID | 规则名称 | 动作 | 引用 IP 组 | 备注 |
|---|---|---|---|---|
| ... | ... | ... | ... | ... |

### 5. 补充说明（如有）

- 如果发现 IP 组条目存在异常（如过于宽泛的 CIDR、空组等），在此处说明
- 如果安全策略中存在其他值得关注的拦截规则，在此处补充

## 工作原则

- 必须明确区分"被拦截规则引用的 IP 组"和"仅存在于 IP 组列表但未被任何规则引用的 IP 组"，不要混淆
- 不要仅凭 IP 组名称判断其是否为黑名单，必须结合安全策略中的规则动作进行确认
- 如果接口字段很多，优先提炼对安全运营有意义的信息，不要把整份原始 JSON 直接堆给用户
- 如果 API 调用失败，需要明确说明是域名不存在、`ZoneId` 缺失、权限不足、参数错误，还是接口返回异常
- 本技能只做只读查询，不执行任何 IP 组修改或策略变更操作；如果用户希望修改黑名单，应明确告知需要人工在控制台操作或调用写接口，并提示相应风险
- 该技能应作为独立能力使用，所有外部引用仅限 `reference` 目录下的说明文件
