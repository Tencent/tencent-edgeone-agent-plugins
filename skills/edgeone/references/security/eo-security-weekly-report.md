# eo-security-weekly-report

用于产出 **EdgeOne 安全防护周报** 或 **当前安全配置快照**。该技能的核心目标是：

- 定期沉淀站点安全配置基线
- 感知策略是否出现异常变更
- 在用户只问“现在的安全配置怎么样”时，快速生成一份可读、可追踪的状态报告

## 适用输入

- 帮我出一份本周的安全防护状态报告
- 查一下现在的安全配置
- 看看某个站点最近有没有安全策略变更
- 帮我整理当前站点的安全策略快照

## 前置条件

### 认证与 CLI 前置动作

1. 所有腾讯云 API 调用统一通过 `tccli` 执行。
2. 如果环境中尚未配置可用凭证，必须先引导用户执行 [auth.md](./reference/auth.md) 中说明的登录方式：

```sh
tccli auth login
```

3. 需要明确告知用户：执行 `tccli auth login` 后，终端会打印授权链接，并在用户完成浏览器授权前保持阻塞；授权成功后命令会自动结束。
4. 严禁向用户索要 `SecretId` / `SecretKey`，也不要执行可能暴露凭证内容的命令（尤其是 tccli configure list）。

### 业务前置条件

1. 在确认站点参数并执行后续查询前，必须先确认 `tccli` 已处于可用登录状态。
2. 在确认 `tccli` 可用后，再确认目标站点的 `ZoneId`。
3. 如果用户只提供了域名，而当前上下文无法唯一定位到站点，应在确认 `tccli` 可用后，再要求用户补充 `ZoneId` 或其他可唯一定位站点的信息。
4. 周报默认按“本周”输出；如果用户没有说明时间范围，需要在最终报告中明确写出起止时间。
5. 同一份报告中的 API 调用应按步骤顺序执行，避免不同接口结果在时间上不一致。

## 执行流程

### 1. 检查认证状态

优先确认当前环境中的 `tccli` 是否已登录且可用。

如果当前环境尚未登录或调用失败提示凭证不可用，先引导用户完成 [auth.md](./reference/auth.md) 中的浏览器授权登录，再继续后续步骤。

### 2. 确认范围

在确认 `tccli` 可用后，再确认本次报告的最小必要信息：

- 站点标识
- `ZoneId`
- 时间范围
- 是否已有可用于对比的历史基线

如果缺少 `ZoneId`，先补齐站点信息，再进入采集阶段。

### 3. 查询接口文档（按需）

如果对 `teo` 相关接口的参数、字段、数据结构或返回含义不确定，应按照 [ref.md](./reference/ref.md) 中的方法先检索文档，再执行 API 调用，不要臆测字段含义。

典型检索方式包括：

- 检索 `teo` 服务的接口列表
- 阅读对应 Action 文档确认参数
- 阅读数据结构文档确认字段含义

### 4. 顺序采集数据

围绕同一个 `ZoneId`，顺序采集以下 3 组数据：

```sh
tccli teo DescribeSecurityPolicy --ZoneId <ZoneId>
tccli teo DescribeWebSecurityTemplates --ZoneId <ZoneId>
tccli teo DescribeSecurityIPGroup --ZoneId <ZoneId>
```

如果接口返回提示参数不完整、权限不足、字段名不匹配或其他异常，应按以下顺序处理：

- 先说明失败原因
- 如果是参数不完整，先执行 `tccli teo <Action> help` 获取该接口所需参数；如果返回信息仍不够清晰，再执行 `tccli teo <Action> help --detail` 获取详细说明
- 如果仍有疑问，再根据 [ref.md](./reference/ref.md) 的方法查询对应接口文档
- 补充必要参数后重新调用

### 5. 生成结论与快照

在完成采集后，先输出结论，再给出支撑证据，并在最后附上可用于后续比较的精简基线快照。

## 数据采集与分析方法

### 1. 安全策略总览

基于 `DescribeSecurityPolicy`：

- 识别当前已启用、关闭或处于观察模式的核心防护能力
- 提炼默认动作、拦截/观察模式、例外放行等关键信息
- 标记高风险配置，例如：
  - 关键防护能力被关闭
  - 策略从拦截降级为观察或关闭
  - 出现范围过大的放行配置

### 2. Web 安全模板情况

基于 `DescribeWebSecurityTemplates`：

- 汇总模板名称、模板 ID、模板类型、绑定情况、最近修改时间（如果返回结果中有）
- 关注本周是否存在新增、解绑、替换或批量调整
- 如果关键模板缺失、未绑定或状态异常，标记为需要人工确认

### 3. 安全 IP Group 情况

基于 `DescribeSecurityIPGroup`：

- 汇总 IP Group 名称、ID、类型、条目数量、用途或引用关系（如果返回结果中有）
- 关注 allow/deny 列表是否为空、是否突然扩大、是否存在测试组长期生效
- 识别未知来源新增网段、异常大网段或明显不符合命名规范的对象

## 变更判断规则

如果能够获取历史基线（例如用户提供的上周报告、当前对话中的旧快照、已有文件中的历史结果或其他明确来源），则执行差异对比，并按以下维度输出：

- 新增项
- 删除项
- 修改项
- 风险升级
- 风险降级

如果**拿不到历史快照**：

- 明确说明“当前仅能生成配置快照，无法单独判断是否属于异常变更”
- 将本次结果作为新的基线快照输出，便于后续周报继续对比

## 输出要求

最终必须用中文输出，优先给结论，再给证据，结构固定如下：

### 1. 报告范围

- 站点标识
- `ZoneId`
- 时间范围
- 数据来源接口

### 2. 当前安全配置摘要

- 核心防护能力状态
- 模板绑定摘要
- IP Group 摘要

### 3. 本周变更与风险

- 本周发现的配置变化
- 疑似异常变更
- 需要人工确认的事项

### 4. 处理建议

- 立即处理项
- 本周内处理项
- 持续观察项

### 5. 附录：原始快照

在结尾附一个**精简 JSON 快照**，至少包含以下字段，便于后续继续对比：

```json
{
  "zoneId": "",
  "generatedAt": "",
  "timeRange": {
    "start": "",
    "end": ""
  },
  "securityPolicy": {},
  "webSecurityTemplates": [],
  "securityIPGroups": []
}
```

## 工作原则

- 不要凭空判断“异常变更”，必须给出与当前配置或历史快照对应的证据
- 用户只问“查一下现在的安全配置”时，也按“当前快照 + 风险提示”的方式回答
- 如果接口字段很多，优先提炼对安全运营有意义的信息，不要把整份原始 JSON 直接堆给用户
- 如果 API 调用失败，需要明确说明是 `ZoneId` 缺失、权限不足、参数错误，还是接口返回异常
- 如果缺少历史基线，不要强行输出“本周变更结论”，应明确写成“本次基线快照”
- 该技能应作为独立能力使用，所有外部引用仅限 `reference` 目录下的说明文件
