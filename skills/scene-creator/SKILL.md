# EdgeOne 场景文档创建器

帮助用户在 EdgeOne skill 仓库中添加新的场景 reference 文档。根据用户描述的场景和 API 文档，编写符合现有风格规范的场景指引，并更新对应模块的 README.md 索引。

> **本 skill 创建的是 references 场景文档，不是完整的 skill。**
> 场景文档位于 `skills/edgeone/references/<module>/` 目录下，是 EdgeOne 主 skill 的一部分。

## ⛔ 强制规则：每一步都必须和用户确认

> **禁止跳步、禁止自行决定。** 整个流程中有 **3 个阻断点**（标记为 ⛔），每个阻断点都必须停下来等用户回复确认后才能继续。
>
> 绝对不允许的行为：
> - 在用户确认场景和子场景之前搜索 API 或读取仓库文件
> - 在用户确认 API 清单之前编写场景文档
> - 在任何阻断点处"猜测用户会同意"而跳过确认

## 工作流总览

```
Step 1: 确认场景需求       ⛔ 阻断点 — 等用户确认
    ↓
Step 2: 细化子场景与模拟情景 ⛔ 阻断点 — 等用户逐一确认每个子场景
    ↓
Step 3: 研究 API 文档       ⛔ 阻断点 — 整理 API 清单后等用户确认
    ↓
Step 4: 起草场景文档（按模板编写 .md 文件）
    ↓
Step 5: 更新模块 README.md 索引
    ↓
Step 6: 用户评审与最终确认（提示用户确认）
```

---

## Step 1：确认场景需求

> **⛔ 阻断点 1**：本步骤结束时必须停下来等待用户回复。在用户回复并确认之前，**不得**进入 Step 2。

本步骤的唯一目标是**向用户提问**。不要读取仓库文件、不要搜索 API、不要做任何准备工作。

使用 `ask_followup_question` 向用户确认以下核心信息：

1. **目标模块**：场景文档放在哪个模块下？（acceleration / security / 其他新模块）
2. **场景名称**：简短的英文标识（用于文件名），如 `set-origin-group`、`dns-record-manager`
3. **场景目标**：一句话说明这个场景帮用户完成什么，如"管理源站组配置"
4. **大致包含哪些子操作**：如"查询、创建、修改、删除"——不需要精确，大致范围即可

> **提问后立即停止**，等待用户回复。不要在同一轮中继续执行任何操作。

### 参考：现有模块列表

现有模块（可能随时间变化，以实际目录为准）：

| 模块目录 | 说明 |
|---|---|
| `acceleration/` | 站点加速：站点接入、缓存、证书等 |
| `security/` | 安全防护：策略盘查、模板审计、黑名单等 |
| `api/` | API 调用基础设施（通常不在此添加场景） |

如果用户的场景不属于现有模块，需讨论是否创建新模块目录。

---

## Step 2：细化子场景与模拟情景

> **⛔ 阻断点 2**：本步骤结束时必须停下来等待用户**逐一确认每个子场景**。在用户确认之前，不得搜索 API 或编写文档。

根据 Step 1 用户给出的大致操作范围，将场景拆分为具体的子场景，并为**每个子场景**提供模拟情景，让用户直观理解 agent 未来的行为。

### 2.1 列出所有子场景

为每个子场景给出：

- **子场景编号和标题**：如"场景 A：查询现有源站组"
- **触发短语**：用户会怎么说（2-3 个示例）
- **风险等级**：🟢 低风险（读操作） / 🟡 中风险（非破坏性写操作） / 🔴 高风险（删除、覆盖类操作）

### 2.2 为每个子场景编写模拟情景

**这是最关键的部分。** 为每个子场景写一个"用户 ↔ agent"的完整对话模拟，让用户清楚看到未来 agent 按照这个文档执行时的具体行为。

**模拟情景格式**：

```
📌 场景 A：<子场景标题>

用户说："<一句典型的触发语>"

Agent 执行：
1. <第一步具体操作>
2. <第二步具体操作>
3. ...

Agent 输出示例：
> <agent 实际会给用户看到的输出内容，如表格、摘要等>
```

**示例**（以源站组管理为例）：

```
📌 场景 A：查询现有源站组

用户说："帮我看看有哪些源站组"

Agent 执行：
1. 调用 DescribeOriginGroup 获取源站组列表
2. 将结果整理为表格展示给用户

Agent 输出示例：
> 您的站点下共有 3 个源站组：
>
> | 源站组名称 | 源站组 ID | 源站类型 | 源站地址 |
> |---|---|---|---|
> | primary-group | og-xxxxx1 | IP | 1.2.3.4, 5.6.7.8 |
> | backup-group | og-xxxxx2 | 域名 | origin.example.com |
> | s3-group | og-xxxxx3 | 对象存储 | bucket.cos.region.myqcloud.com |

---

📌 场景 B：创建新源站组

用户说："帮我创建一个源站组"

Agent 执行：
1. 询问用户源站组名称、源站类型、源站地址等必要信息
2. ⚠️ 向用户确认："即将创建源站组 xxx，源站地址为 xxx，确认执行？"
3. 用户确认后，调用 CreateOriginGroup 创建源站组
4. 展示创建结果

Agent 输出示例：
> ⚠️ 即将创建源站组，请确认以下信息：
> - 源站组名称：primary-group
> - 源站类型：IP
> - 源站地址：1.2.3.4（权重 100）、5.6.7.8（权重 50）
>
> 确认执行吗？
```

### 2.3 向用户展示并逐一确认

将上述所有子场景和模拟情景一起展示给用户，然后明确提问：

1. 子场景是否**有遗漏**或**需要删减**？
2. 每个模拟情景中 agent 的行为是否**符合预期**？
3. 输出格式是否**合适**？
4. 触发短语是否**准确**？

> **提问后立即停止**，等待用户对每个子场景逐一反馈。用户可能会要求增删子场景、修改触发短语、调整 agent 行为等。根据反馈调整后，再次向用户确认，直到用户表示满意。

## Step 3：研究 API 文档

> **前置检查**：进入本步骤前，确认已获得用户对以下内容的明确确认：
> - ✅ 目标模块 + 场景名称 + 场景目标
> - ✅ 所有子场景的标题、触发短语、模拟情景
>
> 如果缺少任何一项，**返回对应步骤继续确认**。

通过 EdgeOne 的 cloudcache API 文档源检索场景涉及的所有 API。

### 3.1 检索相关 API

如果用户已提供 API 名称，直接查阅文档：

```sh
curl -s https://cloudcache.tencentcs.com/capi/refs/service/teo/action/<ActionName>.md
```

如果用户只描述了场景但不确定具体 API，使用兜底检索：

```sh
# 搜索接口列表
curl -s https://cloudcache.tencentcs.com/capi/refs/service/teo/actions.md \
  | grep -i "<关键词>"

# 搜索最佳实践
curl -s https://cloudcache.tencentcs.com/capi/refs/service/teo/practices.md \
  | grep -i "<关键词>"
```

### 3.2 整理 API 清单

为每个涉及的 API 记录：

- **Action 名称**：如 `CreateOriginGroup`
- **一句话说明**：如"创建源站组"
- **是否为写操作**：Create\* / Modify\* / Delete\* / Bind\* / Apply\* 等均为写操作
- **关键参数**：必填参数和常用可选参数
- **响应中的关键字段**：需要向用户展示或在后续步骤中使用的字段

### 3.3 阅读数据结构

对于复杂参数或响应中的嵌套结构，进一步查阅：

```sh
curl -s https://cloudcache.tencentcs.com/capi/refs/service/teo/model/<ModelName>.md
```

### 3.4 向用户展示并确认 API 清单

> **⛔ 阻断点 3**：整理完 API 清单后必须停下来等用户确认。在用户确认之前，不得编写场景文档。

向用户展示以下信息：

1. **API 清单总表**：每个 API 的名称、说明、读/写属性
2. **子场景 ↔ API 映射**：每个子场景分别会调用哪些 API，调用顺序是什么
3. **关键参数摘要**：每个 API 的必填参数和重要可选参数

**展示格式**：

```
### API 清单

| Action | 说明 | 读/写 | 对应子场景 |
|---|---|---|---|
| DescribeOriginGroup | 查询源站组列表 | 读 | 场景 A |
| CreateOriginGroup | 创建源站组 | 写 | 场景 B |
| ModifyOriginGroup | 修改源站组 | 写 | 场景 C |
| DeleteOriginGroupConditions | 删除源站组 | 写 | 场景 D |

### 各子场景的 API 调用流程

**场景 A：查询现有源站组**
→ DescribeOriginGroup（必填参数：ZoneId）

**场景 B：创建新源站组**
→ CreateOriginGroup（必填参数：ZoneId、OriginGroupName、Type、Records）
```

明确提问：
1. API 覆盖是否**完整**？有没有遗漏的接口？
2. 各子场景的 API 调用顺序是否**正确**？
3. 是否需要调整某个子场景使用的 API？

> **提问后立即停止**，等待用户确认。

## Step 4：起草场景文档

> **前置检查**：进入本步骤前，确认已获得用户对以下内容的明确确认：
> - ✅ 场景需求（Step 1）
> - ✅ 子场景与模拟情景（Step 2）
> - ✅ API 清单与调用流程（Step 3）
>
> 如果缺少任何一项，**返回对应步骤继续确认**。

### 4.1 阅读现有文档学习风格

在编写前，**必须**读取目标模块下至少一个现有场景文档作为风格参考：

```
skills/edgeone/references/<module>/    → 列出现有文件
```

选择一个复杂度相近的文档通读，重点学习：

- 标题与一句话描述的写法
- 场景划分与命名（场景 A / B / C …）
- 触发短语的风格
- 安全提示的位置和措辞
- 输出格式建议的写法

### 4.2 按模板编写

在 `skills/edgeone/references/<module>/` 下创建 `<scene-name>.md`。

**文件结构模板**（参考 `assets/scene-template.md`）：

```markdown
# <场景标题>

<一句话描述：做什么、覆盖哪些操作>

## 前置条件

1. tccli 环境就绪（参考 [../api/README.md](../api/README.md)）
2. 已获取 ZoneId（参考 [../api/zone-discovery.md](../api/zone-discovery.md)）
3. <场景特有前置条件（如有）>

## 场景 A：<子场景标题>

**触发**：用户说"<触发短语 1>"、"<触发短语 2>"。

<流程步骤说明>

> **（写操作必须有此提示）**：<操作名称>会<影响说明>。**必须**向用户说明操作内容及影响，等待用户明确确认后才能执行。

## 场景 B：<子场景标题>

...

## 输出格式（如有必要）

<建议的输出模板>
```

### 4.3 编写规范

遵循以下规范（与现有文档保持一致）：

- **写作风格**：使用祈使句 / 不定式（动词开头），不使用第二人称
- **只列 API 名称，不写完整命令示例**：具体参数通过 api-discovery.md 查阅
- **写操作安全提示**：每个涉及写操作（Create\* / Modify\* / Delete\* / Bind\* / Apply\*）的场景都**必须**在调用前加安全警告块，说明影响并要求用户确认
- **场景编号**：使用"场景 A / B / C …"格式，子步骤使用"A1 / A2 …"
- **输出建议**：告诉 agent 以什么格式向用户展示结果（表格、摘要等）
- **不要包含 YAML frontmatter**：场景文档是 reference 文件，不是独立 skill，不需要 `---` frontmatter 块
  - 例外：如果目标模块的现有文档有 frontmatter，则保持一致

> **关于是否包含 frontmatter**：以目标模块中现有文档的实际风格为准。
> `acceleration/` 模块的文档不含 frontmatter，`security/` 模块的文档包含 frontmatter。

## Step 5：更新模块 README.md 索引

编写完场景文档后，**必须**更新对应模块的 `README.md`，将新场景添加到索引表格中。

### 5.1 读取现有 README.md

```
plugins/edgeone/references/<module>/README.md
```

### 5.2 添加索引条目

在 `## 本目录文件` 表格中追加一行。不同模块的表格格式略有差异，以实际为准：

**acceleration 风格**（三列）：

```markdown
| [<scene-name>.md](<scene-name>.md) | <风险等级> | 「<触发短语 1>」「<触发短语 2>」 |
```

**security 风格**（三列 + 快速决策树）：

```markdown
| [<scene-name>.md](<scene-name>.md) | <风险emoji> <风险等级> | <核心触发场景一句话描述> |
```

> 同时检查 README.md 中是否有决策树、前置条件等段落需要相应更新。

### 5.3 检查主 SKILL.md 是否需要更新

如果新场景属于现有模块，通常**不需要**修改 `skills/edgeone/SKILL.md`（主入口）。

但如果创建了**新模块目录**，则需要在主 SKILL.md 的「模块入口」表格中添加对应行。

---

## Step 6：用户评审与最终确认

将编写好的场景文档和 README.md 变更展示给用户评审，提示用户确认。重点关注：

1. **场景覆盖完整性**：是否遗漏了用户需要的子场景
2. **触发短语准确性**：是否能准确匹配用户的自然语言表述
3. **API 调用顺序**：流程步骤是否合理，前后依赖是否正确
4. **安全提示**：写操作是否都有对应的安全确认提示
5. **风格一致性**：与同模块下其他文档的写法是否统一

根据用户反馈修改文档，重复评审直到用户满意。

### 最终 checklist

- [ ] 场景文档已创建在正确的模块目录下
- [ ] 文档风格与同模块其他文档一致
- [ ] 所有写操作都有安全确认提示
- [ ] 模块 README.md 索引已更新
- [ ] （如创建新模块）主 SKILL.md 模块入口表格已更新
- [ ] 用户已确认内容无误
